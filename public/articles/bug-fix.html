Fixing a Race Condition in My Browser Game

How a subtle timing bug in a canvas-based game led me down a rabbit hole of debugging asynchronous JavaScript — and the clean fix that came out of it.

I have been building a small browser-based racing game as a side project — a top-down 2D track where you steer a car and try to beat your lap time. It started as a fun way to practice canvas rendering and keyboard input handling, but it quickly turned into a lesson in asynchronous timing.

The Bug

The issue was subtle. Most of the time, the game ran perfectly. But occasionally — especially when the tab was backgrounded and then brought back into focus — the car would teleport to an unexpected position or clip through a wall. It only happened once in a while, which made it incredibly frustrating to track down.

The root cause turned out to be how I was calculating the delta time between frames. I was using a simple timestamp difference: 

const now = Date.now();
const delta = now - lastFrameTime;
lastFrameTime = now;

When the tab is backgrounded, requestAnimationFrame stops firing. But Date.now() keeps ticking. So when the tab comes back, the next delta could be thousands of milliseconds — far larger than any normal frame — and the car position would jump by a huge amount in a single frame.

The Investigation

I started by adding debug logging to track the delta values. Sure enough, after switching tabs for a few seconds and coming back, I would see deltas like 4523ms instead of the usual 16ms. The physics engine was moving the car based on that inflated delta, launching it through walls.

My first instinct was to clamp the delta to a maximum value: 

const delta = Math.min(now - lastFrameTime, 100);

This worked — the car no longer teleported. But it felt like a band-aid. The real problem was that the game loop should not be updating at all when the tab is not visible.

The Clean Fix

The proper solution was to detect when the tab regains focus and reset the last frame time. I used the visibilitychange event: 

document.addEventListener("visibilitychange", () => {
  if (document.visibilityState === "visible") {
    lastFrameTime = Date.now();
  }
});

This ensures that when the player returns to the tab, the next frame starts with a fresh, accurate delta. No more teleporting, no more wall clipping.

What I Learned

This was a good reminder that requestAnimationFrame and time-based physics need careful handling. The browser will pause animation frames when a tab is hidden, but your time source does not pause with it. Always account for the possibility of large time gaps — either by clamping or by resetting when focus returns.

It is a small fix, but it is the kind of thing that separates a polished game from a buggy one. I am glad I took the time to understand the root cause instead of just patching the symptom.
